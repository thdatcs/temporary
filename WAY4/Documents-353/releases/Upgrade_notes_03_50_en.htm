Accounting

CARDS-65930

Changed custom fee rounding behavior for custom fees when one of the accounts is of high accuracy (HP) and the other is a regular one.
Upgrade Note:
The changes have been done for the custom fees with the following settings:
- the tag PRECISION=<N>; is set in service details
- the fee is posted between HP and regular non-HP account.
Old behavior:
The fee amount was rounded to the currency exponent, so for the transfer between HP account and XF HP account rounded amount was used.
New behavior:
The amount for the transfer between HP account and XF HP account is posted according to the precision defined on service.
The transfer between regular non-HP account and non-HP XF account is rounded to currency exponent.
To get the previous behavior, for the custom fee the PRECISION=<N>; tag should be removed. For the cases when High Precision fee amount should be used for other custom fees (SUBTRACT_FEE/ADD_FEE tags) or saved in document (SAVE_FEE tag) - the PRECISION=<N>; tag should be present and XF_ROUND; tag should be set.

CARDS-67812

Fixed incorrect logging in the process message that some records were successfully processed by ext_fx_rates_import pipe in case one of the records inside file led to error generation.
Upgrade Note:
The pipe algorithm has been changed to two phases: on the first step pipe checks possible errors (format, data) in the file being loaded and then on the second step imports data in data base from the file. This logic is applied to each imported file.
It is recommended to test the pipe running time.

Corrected acquiring authorization processing for the Outward Status that was already sent to a payment system SMS interface. Such documents will have the "Sent" outward status after document processing and will not be sent to the IPS a second time.
Upgrade Note:
Clients who use non-standard SMS channel codes must set the additional global parameter CHANNEL_MAPPING. The parameter value format is <standard channel symbol 1><using channel symbol 1><standard channel symbol 2><using channel symbol 2>...
Standard SMS channel codes are S - VISA SMS, K - VISA SMS NSPK, D - Mastercard SMS, d - Mastercard SMS NSPK, Y - AMEX SMS, L - DCI SMS, f - NSPK SMS. For example, CHANNEL_MAPPING="SdYm" means that the channel code "d" is used for VISA SMS channel and "m" - for AMEX SMS channel.

CARDS-66560

Fixed an error when ACQ_ONLINE tag was redefined on service level but redefinition did not work because on transaction subtype level ACQ_ONLINE tag was defined as "A" ("A" value redefines the value of the Is Online field to the opposite value).
Upgrade Note:
Corrected the behavior of the system when determining the online attribute responsible for sending a transactional message in online mode. 
The ACQ_ONLINE tag specified at the service level will override the value of the ACQ_ONLINE tag at the transaction sub-type level.
Before installing new WAY4Cards version it is necessary to analyze value of  ACQ_ONLINE tag at the transaction sub-type level and at the service level, especially if both of them are set.
Particular attention should be paid if the ACQ_ONLINE tag is set to A.
If the tag is set at the service level to the value ACQ_ONLINE=A, then you need to check whether this tag is set at  the transaction's sub-type level to the same value (ACQ_ONLINE=A).
Such settings may have side effects, because ONLINE attribute is redefined to the opposite.
It is recommended to perform additional regression testing and adjust the existing settings if this settings are found.
Check Scripts:
The script CARDS-66560_Check_ACQ_ONLINE_On_Service.sql should be used to define configuration to be analyzed.

New Behavior:
CurrentDue = rate*(MTP_AGED_PART-PAST_DUE+MTP_AGED_FULL_PREV[-OVL])+MTP_AGED_FULL

CurrentDue = rate*(MTP_AGED_PART-PAST_DUE+MTP_AGED_FULL_PREV-[OVL])+MTP_AGED_FULL

Main Only=Main Only
History Mode = None
2) For account templates which are defined as Due templates for the account templates marked with MTP_AGED_FULL balance - add the value MTP_AGED_FULL_PREV to the Extra Bal Type field.
Example:
The following account templates and ageing rule for them are defined in the account scheme:
Fee Current->Fee Grace->Fee Billed
The fee is included in MTP as 100%. In the current configuration, the Fee Grace account template is marked with the MTP_AGED_FULL balance. The Fee Billed account template is not marked with the MTP_AGED_FULL balance. In this case, this account template should be marked with MTP_AGED_FULL_PREV balance.

      Upgrade Note:
      1. The script CARDS--65146_BALANCE_TYPE.sql checks a configuration for a MTP_AGED_FULL_PREV balance, which is used to calculate the minimum payment (MTP) according to the BT_SIMPLE algorithm. If the script result is not empty - note that rules for calculating MTP according to the BT_SIMPLE-PAST_DUE[-OVL] formula have been changed.
Old behavior:
CurrentDue = rate*(MTP_AGED_PART-PAST_DUE[-OVL])+MTP_AGED_FULL
New Behavior:
CurrentDue = rate*(MTP_AGED_PART-PAST_DUE+MTP_AGED_FULL_PREV[-OVL])+MTP_AGED_FULL
The reason for the change is that a MTP_AGED_FULL_PREV balance is part of a PAST_DUE balance and the previous behavior was found to be incorrect.
If the MTP_AGED_FULL_PREV balance is used in the configuration, this balance's setup must be checked. This balance type should be defined for the account template that is specified as the Due Template for the account template with the MTP_AGED_FULL balance.
Example:
A line of accounts is configured in an Account Scheme:
Fee Current->Fee Grace->Fee Billed
A fee is included in MTP as 100%. The Fee Grace account should be marked with the MTP_AGED_FULL balance. The Fee Billed account template must be marked with the MTP_AGED_FULL_PREV balance.
2. The script CARDS--65146_ACC_SCHEMES.sql checks the configuration for the mode for calculating MTP according to BT_SIMPLE-PAST_DUE [-OVL]. If this mode is used and there are also account templates marked with the MTP_AGED_FULL balance (i.e. included in MTP as 100%), it is recommended to switch to the new mode for calculation according to the following formula:
CurrentDue = rate*(MTP_AGED_PART-PAST_DUE+MTP_AGED_FULL_PREV-[OVL])+MTP_AGED_FULL
where MTP_AGED_FULL_PREV is the balance that was already (in previous cycles) included in MTP as 100% according to the MTP_AGED_FULL balance but has not been paid yet.
The following changes should be made to the configuration:
1) Create a new balance type MTP_AGED_FULL_PREV in the form "OpenWay\Full\Configuration Setup\Accounting Setup\Balance Types" with the parameters:
Code = MTP_AGED_FULL_PREV
Main Only=Main Only
History Mode = None
2) For account templates which are defined as Due templates for the account templates marked with MTP_AGED_FULL balance - add the value MTP_AGED_FULL_PREV to the Extra Bal Type field.
Example:
The following account templates and ageing rule for them are defined in the account scheme:
Fee Current->Fee Grace->Fee Billed
The fee is included in MTP as 100%. In the current configuration, the Fee Grace account template is marked with the MTP_AGED_FULL balance. The Fee Billed account template is not marked with the MTP_AGED_FULL balance. In this case, this account template should be marked with MTP_AGED_FULL_PREV balance.
3) Recalculate the MTP_AGED_FULL_PREV balance for existing contracts by using "OpenWay\Full\Configuration Setup\Accounting Setup\Balance Types"->Recalc Balance For All Contracts.

Old behavior:
CurrentDue = rate*(MTP_AGED_PART-PAST_DUE[-OVL])+MTP_AGED_FULL
New Behavior:
CurrentDue = rate*(MTP_AGED_PART-PAST_DUE+MTP_AGED_FULL_PREV[-OVL])+MTP_AGED_FULL
The reason for the change is that a MTP_AGED_FULL_PREV balance is part of a PAST_DUE balance and the previous behavior was found to be incorrect.

Example:
A line of accounts is configured in an Account Scheme:
Fee Current->Fee Grace->Fee Billed
A fee is included in MTP as 100%. The Fee Grace account should be marked with the MTP_AGED_FULL balance. The Fee Billed account template must be marked with the MTP_AGED_FULL_PREV balance.
2. The script CARDS--65146_ACC_SCHEMES.sql checks the configuration for the mode for calculating MTP according to BT_SIMPLE-PAST_DUE [-OVL]. If this mode is used and there are also account templates marked with the MTP_AGED_FULL balance (i.e. included in MTP as 100%), it is recommended to switch to the new mode for calculation according to the following formula:
CurrentDue = rate*(MTP_AGED_PART-PAST_DUE+MTP_AGED_FULL_PREV-[OVL])+MTP_AGED_FULL
where MTP_AGED_FULL_PREV is the balance that was already (in previous cycles) included in MTP as 100% according to the MTP_AGED_FULL balance but has not been paid yet.
The following changes should be made to the configuration:
1) Create a new balance type MTP_AGED_FULL_PREV in the form "OpenWay\Full\Configuration Setup\Accounting Setup\Balance Types" with the parameters:
Code = MTP_AGED_FULL_PREV
Main Only=Main Only
History Mode = None
2) For account templates which are defined as Due templates for the account templates marked with MTP_AGED_FULL balance - add the value MTP_AGED_FULL_PREV to the Extra Bal Type field.
Example:
The following account templates and ageing rule for them are defined in the account scheme:
Fee Current->Fee Grace->Fee Billed
The fee is included in MTP as 100%. In the current configuration, the Fee Grace account template is marked with the MTP_AGED_FULL balance. The Fee Billed account template is not marked with the MTP_AGED_FULL balance. In this case, this account template should be marked with MTP_AGED_FULL_PREV balance.
3) Recalculate the MTP_AGED_FULL_PREV balance for existing contracts by using "OpenWay\Full\Configuration Setup\Accounting Setup\Balance Types"->Recalc Balance For All Contracts.

If the script result is not empty - note that rules for calculating MTP according to the BT_SIMPLE-PAST_DUE[-OVL] formula have been changed.
Old behavior:
CurrentDue = rate*(MTP_AGED_PART-PAST_DUE[-OVL])+MTP_AGED_FULL
New Behavior:
CurrentDue = rate*(MTP_AGED_PART-PAST_DUE+MTP_AGED_FULL_PREV[-OVL])+MTP_AGED_FULL
The reason for the change is that a MTP_AGED_FULL_PREV balance is part of a PAST_DUE balance and the previous behavior was found to be incorrect.
If the MTP_AGED_FULL_PREV balance is used in the configuration, this balance's setup must be checked. This balance type should be defined for the account template that is specified as the Due Template for the account template with the MTP_AGED_FULL balance.
Example:
A line of accounts is configured in an Account Scheme:
Fee Current->Fee Grace->Fee Billed
A fee is included in MTP as 100%. The Fee Grace account should be marked with the MTP_AGED_FULL balance. The Fee Billed account template must be marked with the MTP_AGED_FULL_PREV balance.
2. The script CARDS--65146_ACC_SCHEMES.sql checks the configuration for the mode for calculating MTP according to BT_SIMPLE-PAST_DUE [-OVL]. If this mode is used and there are also account templates marked with the MTP_AGED_FULL balance (i.e. included in MTP as 100%), it is recommended to switch to the new mode for calculation according to the following formula:
CurrentDue = rate*(MTP_AGED_PART-PAST_DUE+MTP_AGED_FULL_PREV-[OVL])+MTP_AGED_FULL
where MTP_AGED_FULL_PREV is the balance that was already (in previous cycles) included in MTP as 100% according to the MTP_AGED_FULL balance but has not been paid yet.
The following changes should be made to the configuration:
1) Create a new balance type MTP_AGED_FULL_PREV in the form "OpenWay\Full\Configuration Setup\Accounting Setup\Balance Types" with the parameters:
Code = MTP_AGED_FULL_PREV
Main Only=Main Only
History Mode = None
2) For account templates which are defined as Due templates for the account templates marked with MTP_AGED_FULL balance - add the value MTP_AGED_FULL_PREV to the Extra Bal Type field.
Example:
The following account templates and ageing rule for them are defined in the account scheme:
Fee Current->Fee Grace->Fee Billed
The fee is included in MTP as 100%. In the current configuration, the Fee Grace account template is marked with the MTP_AGED_FULL balance. The Fee Billed account template is not marked with the MTP_AGED_FULL balance. In this case, this account template should be marked with MTP_AGED_FULL_PREV balance.
