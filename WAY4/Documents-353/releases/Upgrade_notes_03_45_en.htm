CARDS-56959

Supported 3 logging modes for the CS_DECISION_LOG table. Default behaviour has been changed.
Upgrade Note:
Changes have been made that impact configurations using Decisions.

To check for the existence of such a configuration, use the standard form: OpenWay\Full\Configuration Setup\Common Handbooks\Decision.
If it is not empty, pay careful attention to the information below.

By default, logging of changes for all Decisions to the CS_DECISION_LOG table is disabled. This does cause any changes in processing, as this log is not used in processing. The current value of a Decision is always recalculated on the fly at the time it is asked.

Previously, logging of changes in Decisions was always deferred, using Object Task Scheduler, which sometimes resulted in a large queue of tasks for recalculating and logging the values of Decisions, although the CS_DECISION_LOG may not have been not used anywhere.

Disabling logging does not concern Decisions that are used in Datamart export. These decisions are always logged, regardless of any parameters. Only two log mode are possible for these Decisions: Deferred or Immediate (default).

Three log modes are now supported in WAY4:
1. SKIP - Don't log anything. Used by default for all Decisions, except those exported into Datamart.
2. DEFERRED - Deferred logging in CS_DECISION_LOG using Object Task Scheduler. This logging mode corresponds to the system's previous behaviour. Make sure that Object Task Scheduler is run regularly before enabling deferred log mode.
3. IMMEDIATE - Immediately recalculation and saving in CS_DECISION_LOG when a linked Classifier changes. This is used as the default mode for Decisions exported to Datamart. This mode is not recommended for Decisions that can be used for large contract trees, since a change in the value of a classifier in a hierarchy's top contract may cause immediate recalculation of dependent Decisions for all contracts in the hierarchy.

Log mode can be changed with the global parameter DECISION_LOG_MODE.
For the same behaviour for all Decisions (except those exported to Datamart), one of the aforementioned values should be used: SKIP or DEFERRED or IMMEDIATE.

To check logging mode for each Decision separately, use the value DECISION_LOG_MODE=CHECK_<Default Mode>, where Default Mode is a value: SKIP, DEFERRED, or IMMEDIATE. If the LOG_MODE tag is not set in a specific Decision, default mode will be used.

To set log mode for a Decision, use the LOG_MODE tag in the "Add Info" field. The mode in a specific decision is only checked when the global parameter has the value CHECK_<Default mode> and for Decisions used for export to Datamart.

If the CS_DECISION_LOG table is used in exports or custom processes, log mode should be changed according to these recommendations.

CARDS-51689

MIN_BILLING parameter default behaviour has been changed.
Upgrade Note:
The behaviour of the MIN_BILLING parameter has changed. This parameter is set as a tag value in days in an Accounting Scheme or as the Min Billing parameter in Billing Scheme tariffs.
Earlier it was necessary to set value to one day less than the required minimum length of a billing cycle. For example, to limit the minimum length of a cycle to 28 days it was necessary to set MIN_BILLING=27. Now the actual value, MIN_BILLING=28 must be set. Accordingly, the parameter's behaviour now corresponds to documentation.
The change only affects values set as days. Values set in months do not have this problem.
If a client's configuration uses the MIN_BILLING parameter in Accounting Schemes or in Billing Scheme tariffs, the preliminary phase of upgrade will show a message that a problematic configuration exists. Additional details can be found in Process Messages.
In this case, it is necessary to analyse if a shift of minus one day was used in existing settings. If so, the value must be corrected by adding one day.
Values should be corrected after the upgrade process is finished but before the first Contract Daily Update process.
For more information, contact WAY4 support.

CARDS-57867 / NET-140

Changed the algorithm for creating client/contract address lists and logic for transliterating addresses so that addresses are no longer shown multiple times in WAY4 Web screens and in DB Manager/WAY4 Manager forms. See Client Upgrade Notes for additional information.
Upgrade Note:
The algorithm for defining a language for transliterating client or contract addresses has been changed so that addresses are no longer shown multiple times in WAY4 Web address screens and in the "Customer Service Screen" form in DB Manager/WAY4 Manager. 
Previously, to transliterate an address to English, WAY4 used a language specified for the client in case client address was used for the address_type by default (Default Address="Client Base Address" in "ROOT\OpenWay\Full\Configuration Setup\Client Classifiers\Address Types") and there was no address with such address type in the CLIENT_ADDRESS table, 
or in case there was no CLIENT_ADDRESS at all (address was specified only in Client record) . 
The new behaviour is supported by additional global parameter USE_SYSTEM_ADDR_LANG with the "Y" value (default). 
Now WAY4 does not take client language into account, and in the aforementioned case uses the default system language from Global Constants. If language is not set for client or contract address, or it is set to English, or officer language is not defined, no transliteration for this address is performed.
The behaviour can be switched to use client language with additional global parameter USE_SYSTEM_ADDR_LANG set to "N" in case client address was used for the address_type by default and there was no address with such address type in the CLIENT_ADDRESS table, 
or in case there is no CLIENT_ADDRESS at all.

Now the client address list does not contain address specified in client record. The behaviour can be switched to the previous behaviour using additional global parameter USE_CL_BASE_ADDR with the "Y" value.

Logic for transliterating addresses has been fixed:
1. If "Date To" of one address record equals to "Date From" of another address record with the same address type, WAY4 shows the latest created address record.
2. If the activation period of one address record overlaps the activation period of another address record with the same address type, WAY4 shows the address with fewer activation period.

      Upgrade Note:
      The algorithm for defining a language for transliterating client or contract addresses has been changed so that addresses are no longer shown multiple times in WAY4 Web address screens and in the "Customer Service Screen" form in DB Manager/WAY4 Manager. 
Previously, to transliterate an address to English, WAY4 used a language specified for the client in case client address was used for the address_type by default (Default Address="Client Base Address" in "ROOT\OpenWay\Full\Configuration Setup\Client Classifiers\Address Types") and there was no address with such address type in the CLIENT_ADDRESS table, 
or in case there was no CLIENT_ADDRESS at all (address was specified only in Client record) . 
The new behaviour is supported by additional global parameter USE_SYSTEM_ADDR_LANG with the "Y" value (default). 
Now WAY4 does not take client language into account, and in the aforementioned case uses the default system language from Global Constants. If language is not set for client or contract address, or it is set to English, or officer language is not defined, no transliteration for this address is performed.
The behaviour can be switched to use client language with additional global parameter USE_SYSTEM_ADDR_LANG set to "N" in case client address was used for the address_type by default and there was no address with such address type in the CLIENT_ADDRESS table, 
or in case there is no CLIENT_ADDRESS at all.

Now the client address list does not contain address specified in client record. The behaviour can be switched to the previous behaviour using additional global parameter USE_CL_BASE_ADDR with the "Y" value.

Logic for transliterating addresses has been fixed:
1. If "Date To" of one address record equals to "Date From" of another address record with the same address type, WAY4 shows the latest created address record.
2. If the activation period of one address record overlaps the activation period of another address record with the same address type, WAY4 shows the address with fewer activation period.

By default, logging of changes for all Decisions to the CS_DECISION_LOG table is disabled. This does cause any changes in processing, as this log is not used in processing. The current value of a Decision is always recalculated on the fly at the time it is asked.

Previously, logging of changes in Decisions was always deferred, using Object Task Scheduler, which sometimes resulted in a large queue of tasks for recalculating and logging the values of Decisions, although the CS_DECISION_LOG may not have been not used anywhere.

Disabling logging does not concern Decisions that are used in Datamart export. These decisions are always logged, regardless of any parameters. Only two log mode are possible for these Decisions: Deferred or Immediate (default).

Three log modes are now supported in WAY4:
1. SKIP - Don't log anything. Used by default for all Decisions, except those exported into Datamart.
2. DEFERRED - Deferred logging in CS_DECISION_LOG using Object Task Scheduler. This logging mode corresponds to the system's previous behaviour. Make sure that Object Task Scheduler is run regularly before enabling deferred log mode.
3. IMMEDIATE - Immediately recalculation and saving in CS_DECISION_LOG when a linked Classifier changes. This is used as the default mode for Decisions exported to Datamart. This mode is not recommended for Decisions that can be used for large contract trees, since a change in the value of a classifier in a hierarchy's top contract may cause immediate recalculation of dependent Decisions for all contracts in the hierarchy.

Log mode can be changed with the global parameter DECISION_LOG_MODE.
For the same behaviour for all Decisions (except those exported to Datamart), one of the aforementioned values should be used: SKIP or DEFERRED or IMMEDIATE.

To check logging mode for each Decision separately, use the value DECISION_LOG_MODE=CHECK_<Default Mode>, where Default Mode is a value: SKIP, DEFERRED, or IMMEDIATE. If the LOG_MODE tag is not set in a specific Decision, default mode will be used.

To set log mode for a Decision, use the LOG_MODE tag in the "Add Info" field. The mode in a specific decision is only checked when the global parameter has the value CHECK_<Default mode> and for Decisions used for export to Datamart.

If the CS_DECISION_LOG table is used in exports or custom processes, log mode should be changed according to these recommendations.
