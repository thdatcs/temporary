CARDS-48377 / I1603050966

Fixed problem with GL Entry being generated but not included in the GL Entries export file. This problem is rare and caused by incorrect GL configuration (empty HD GL Number in Interbranch account templates). Now the posting process will stop with a fatal error instead of generating incorrect GL Entries with empty debit GL account or credit GL account fields. For more details, see the "Upgrade notes on Impact changes" section.
Upgrade Note:
Исправлена проблема, когда GL-проводки формировались, но не попадали в файл выгрузки проводок. Эта проблема встречается редко и связана с неправильной конфигурацией Главной Книги (пустые значения в поле HeadOffice GL # для шаблонов счетов в Схемах Счетов Interbranch Loro/Nostro или пустые значения в поле GL # в прочих шаблонах счетов). Теперь процесс обработки макротранзакций при попытке сформировать GL-проводку с пустыми значениями  Debit account number или Credit account Number будет остановлен с критической ошибкой.
Необходимо выполнить следующий проверочный запрос, чтобы убедиться, что этой проблемы не существовало: 

select count(*) from gl_transfer where order_status = 'A' and outward_status is null and (cr_number is null or dr_number is null)

Если запрос вернул ненулевое значение, необходимо связаться со службой поддержки поставщика системы WAY4 для получения дальнейших инструкций.
При обнаружении первой GL-проводки с пустым номером счета по дебету или по кредиту, процедура обработки макротранзакций будет прервана, таким образом даже одна некорректная проводка может заблокировать весь процесс обработки. Чтобы продолжить обработку остальных GL-проводок, пока проблемная анализируется, рекомендуется установить параметр процесса EXCEPTION_LIMIT для процесса обработки (подробнее см. главу "Параметры процессов" в документе Global_Parameters.pdf)

Application management

CARDS-49075

In Advanced Applications R2, application accept is downgraded to approve in secondary nodes.
Upgrade Note:
При работе на Secondary-узле Advanced Application R2 запрос на принятие (Accept) неразрешенных заявлений заменяется на их проверку и утверждение (Approve).
В режиме NoWF, при успешной проверке заявления, формируется предупреждение "Application is approved successfully and will be processed later". 
Для обработки этой ситуации при работе по Workflow необходимо использовать настройку в стратегии: 
- В OpenWay\Advanced Applications R2\Setup\Workflow Configuration\OpenWay Settings\Activity Results добавить запись с кодом 2265 (это код ошибки, что заявление такого типа нельзя обработать на Secondary-узле), 
- Этот результат зарегистрировать для действия Aссept - OpenWay\Advanced Applications R2\Setup\Workflow Configuration\Activities. 
- Добавить правило в Rules, что в случае Accept с данным результатом, надо выполнить действие Approve. 

В результате, завершившаяся с кодом 2265 попытка принятия заявления заменится на его проверку, и отправителю заявления будет возвращено предупреждение "Application is approved successfully and will be processed later", оно же сохранится в истории для заявления.
После миграции таких заявлений на Primary-узел они будет обработаны ближайшим групповым принятием (OpenWay\Advanced Applications R2\Application Processing\Issuing Applications - Accept, Acquiring Applications - Accept)
Настройка добавлена в эталон и выгружена в \WAY4Source\WAY4Cards\Opt\applications_r2\db\datam\StandardWFStrategies.txt

CARDS-49531

"BT_SIMPLE" mode processing has changed: Current Due Min Amount Threshold, configured as specified in the Statement Tariff or Account Scheme Min Repayment value, and Full Statement Balance Threshold, configured as a new Threshold tariff with the predefined code "MTP_FULL_BALANCE", have been introduced.
Upgrade Note:
Изменился метод расчета минимального платежа по методу "BT_SIMPLE", который может потребовать изменений конфигурации. Значение Min Repayment, которое может быть задано с помощью Statement Tariff или на Схеме Счетов, больше не используется, как значение параметра Full Statement Balance Threshold. Согласно новому поведению системы (при условии, что значение Repayment Percent не равно нулю), если сумма текущей минимальной задолженности, выставляемой к погашению, меньше, чем значение Min Repayment, то сумма текущей минимальной задолженности будет увеличена до значения Min Repayment. В качестве параметра Full Statement Balance Threshold необходимо использовать новый Threshold-тариф с кодом "MTP_FULL_BALANCE".

CARDS-50322

GL traces with a date in the past are not written to the exception table that can be viewed in OpenWay\Full\DB Administrator Utilities\Special OpenWay Utilities\GL\GL Trace Exceptions if the "Close GL" process has not been launched.
Upgrade Note:
GL-проводки, датированные вчерашним днем, но созданные в текущем открытом банковском дне, больше не записываются в таблицу исключений OpenWay\Full\DB Administrator Utilities\Special OpenWay Utilities\GL\GL Trace Exceptions, если в этом дне еще не выполнялся процесс закрытия проводок "Close GL". Ранее проводки могли ошибочно быть записаны в таблицу исключений, например, при формировании проводок стандартным процессом начисления резервов по ссудам (OpenWay\Full\Daily Procedures\Reserves Accrual\Reserves Accrual for Contracts in Prev Day). Вне процесса Contracts Daily Update будет выполняться проверка, что отсутствует точка согласованности типа "Close GL" с датой больше, чем дата проводки. Если точки нет, то проводка не регистрируется в "GL Trace Exceptions", в противном случае - регистрируется. Таким образом, все проводки за вчерашний день до запуска процесса закрытия проводок не будут попадать в "GL Trace Exceptions".

CARDS-50665

CARDS-49252

The functionality of Balance Types with the "COUNTER_ONLY" tag in the "Posting Details" field has been fixed for multi-currency cumulation mode (when accounts in different currencies are registered for the contract).
The logic for recording COUNTER_ONLY balances has also been changed, now they are "positive" for Advices and "negative" for Reversals.
A new technical currency - "NUM" has been added for recording "COUNTER_ONLY" Balance Types.
Please see "Client Upgrade Notes" for detailed information.
Upgrade Note:
Изменен алгоритм подсчета количества операций, осуществляемый посредством типа балансов (Balance Type) с использованием тега COUNTER_ONLY (в поле "Posting details"), для случаев когда для контракта зарегистрированы счета в разных валютах. Для этого введена новая техническая валюта для учета количества операций.
Теперь типы балансов "COUNTER_ONLY" учитываются не в валюте контракта, а в технической валюте "NUM", при этом в поле "Currency" указывается не цифровой код, а буквенный (доступ к информации о значениях типов балансов контракта можно получить после нажатия на кнопку [Additional] в окне "Balance for <…>" соответствующего контракта).
Была также изменена логика учета балансов "COUNTER_ONLY" – теперь уведомления о совершении операций (Advice) считаются, как "положительное" значение, а отмены (Reversal), как "отрицательное". Данное изменение касается типов балансов с Direction=Debit / Debit Entry / None, знак для которых после обновления системы, поменяется на противоположный.
Например:
3 торговые (Retail) операции; 2 операции отмены (Reversal); 1 кредитовая операция (Credit)
- для типа баланса с Direction=Debit значение баланса будет равным 1 (+3 - 2);
- для типа баланса с Direction=Debit Entry значение баланса будет равным 3 (+3);
- для типа баланса с Direction=none значение баланса будет равным 2 (+3 - 2 +1).

Эти изменения влияют на отображение данных в стандартных экранных формах, а также в сообщениях SOA, в которых содержатся данные о балансах контрактов. Возможно также влияние на индивидуально настроенные выгрузки данных о балансах контрактов.
