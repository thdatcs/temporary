CARDS-49075

In Advanced Applications R2, application accept is downgraded to approve in secondary nodes.
Upgrade Note:
For Advanced Applications R2 in a Secondary node, a request to accept unapproved applications is replaced with their approval.
In NoWF mode, in this case when an application has been approved successfully, the warning "Application is approved successfully and will be processed later" is returned. To handle this situation when working according to Workflow, the following setup in the strategy must be used:
- a record with the code 2265 must be added to OpenWay\Advanced Applications R2\Setup\Workflow Configuration\OpenWay Settings\Activity Results (error code that this type of application cannot be processed in a Secondary node).
- register this result for the action Accept - OpenWay\Advanced Applications R2\Setup\Workflow Configuration\Activities.
- Add a rule to Rules, that in the event of Accept with this result, the Approve action must be performed.

As a result, an attempt to accept an application that was completed with the code 2265 is replaced with its approval and the warning "Application is approved successfully and will be processed later" will be returned to the application's sender. This warning will be saved in the application's history.
After these applications are replicated to the Primary node, they will be processed by the nearest group accept (OpenWay\Advanced Applications R2\Application Processing\Issuing Applications - Accept, Acquiring Applications - Accept).
Setup has been added to the standard configuration and exported to \WAY4Source\WAY4Cards\Opt\applications_r2\db\datam\StandardWFStrategies.txt.

CARDS-50322

GL traces with a date in the past are not written to the exception table that can be viewed in OpenWay\Full\DB Administrator Utilities\Special OpenWay Utilities\GL\GL Trace Exceptions if the "Close GL" process has not been launched.
Upgrade Note:
GL traces created within current day, but having GL date in the past are not written to the exception table that can be viewed in OpenWay\Full\DB Administrator Utilities\Special OpenWay Utilities\GL\GL Trace Exceptions if the "Close GL" process has not been launched. Previously, normal GL entries sometimes were erroneousely registered in GL Trace Exceptions table, for example, entries generated by "Bad Debt Reserving" standard process (OpenWay\Full\Daily Procedures\Reserves Accrual\Reserves Accrual for Contracts in Prev Day). To support this new behaviour we added creation of a consistency point of the special type in the mp_consist_point table: the first time the "Close GL" process is run during the day, a point is created with the current date. Outside the Contracts Daily Update process a check will be made that there is no "Close GL" consistency point with a date later than the posting date. If there is no such point, the entry is not registered in "GL Trace Exceptions", otherwise the entry is registered. Thereby, all entries for the previous day will not go into "GL Trace Exceptions", before running "Close GL" process.

CARDS-49252

The functionality of Balance Types with the "COUNTER_ONLY" tag in the "Posting Details" field has been fixed for multi-currency cumulation mode (when accounts in different currencies are registered for the contract).
The logic for recording COUNTER_ONLY balances has also been changed, now they are "positive" for Advices and "negative" for Reversals.
A new technical currency - "NUM" has been added for recording "COUNTER_ONLY" Balance Types.
Please see "Client Upgrade Notes" for detailed information.
Upgrade Note:
Changed the algorithm for calculating the number of transactions made through a Balance Type using the COUNTER_ONLY tag (in the "Posting details" field) for cases when accounts in different currencies are registered for the contract. To do so, a new technical currency has been implemented for recording the number of transactions.
Now "COUNTER_ONLY" Balance Type are recorded in the "NUM" technical currency, not in the contract currency and a letter code, not a number code is specified in the "Currency" field (for access to information about a contract's Balance Type values, click the [Additional] button in the "Balance for <…>" window of the corresponding contract.)
The logic for recording "COUNTER_ONLY" balances has also changed , now Advices are considered "positive" and Reversals as "negative". This change affects Balance Types with Direction=Debit / Debit Entry / None, whose sign will change to the opposite after system upgrade.
For example:
3 Retail transactions; 2 Reversals; 1 Credit transaction
- for a Balance Type with Direction=Debit, the balance value will be equal to 1 (+3 - 2)
- for a Balance Type with Direction=Debit Entry, the balance value will be equal to 3 (+3)
- for a Balance Type with Direction=none, the balance value will be equal to 2 (+3 - 2 +1).
These changes impact how data are shown in standard screen forms and in SOA messages that contain contract balance data. Custom export of contract balance data may also be affected. Standard batch export interfaces are not affected by this change since they do not export data from contract balance tables.

Upgrade Note:
The "SAVE_GL_N" parameter in Accounting Scheme templates is no longer supported and is not processed in WAY4 (OpenWay\Full\Configuration Setup\Products\Issuing Private Products\Private Issuing Account Schemes - [Details] - "Special Parms" field).

- when "GL Number" / "HeadOffice GL Number" change in the Parent Scheme, values are not inherited by a Child Scheme.
- if the account template's "GL Number" / "HeadOffice GL Number" field is empty when reapproving a Child Scheme, values are inherited from the Parent Scheme.

CARDS-52803
A new global parameter AUTH_AMOUNT_DIFF_PCNT_RETAIL has been added. The new parameter is used for all transactions except those with hotel, car rental or airline MCC codes, for which the existing parameter AUTH_AMOUNT_DIFF_PCNT is used. This parameter can set the maximum difference between authorization and financial presentment amounts as a percentage. If the difference is below this value, the whole blocked amount will be unblocked, otherwise only the financial amount will be unblocked.
Upgrade Note:
Starting with version 03.44, the algorithm for unblocking authorization amounts during financial document processing has been changed. A new global parameter AUTH_AMOUNT_DIFF_PCNT_RETAIL has been added that sets the maximum difference between authorization and financial presentment amounts as a percentage. If the difference is exceeds this value, when processing a financial document, the corresponding error message will be generated and the difference between the authorization amount and financial presentment amount will be blocked.
The parameter is used for all transactions, with the exception of transactions with several financial documents for one authorization (for example in the case of car rental, hotel reservations). The global parameter AUTH_AMOUNT_DIFF_PCNT is used for these exceptions.
The aforementioned behaviour impacts document processing in issuer systems. To restore previous system behaviour when the whole authorization amount was blocked, set the global parameter AUTH_AMOUNT_DIFF_PCNT_RETAIL a value allowing a 100% difference between authorization and financial presentment amounts.

CARDS-54454
Upgrade Note:
To support payment system requirements for VISA Fast Funds and MasterCard MoneySend, and to make payments to MIR payment system cards from Russian state institutions, WAY4 can be configured for fast transfer of funds. The use of these settings is determined by the global parameters VISA_FAST_FUNDS, MC_FAST_FUNDS, and MIR_FAST_FUNDS with the "Y" value in the form OpenWay\Full\Configuration Setup\Main Tables\Additional Global Parameters. 
The mechanism for processing fast transfers has changed. 
Now when a document is imported from a payment system clearing file, as before, a document in WAY4 is generated with the transaction type Notif Prs Credit Item. Further, if the original financial document generated online from an authorization is found when the clearing document is processed by Document Processing, the Notif Prs Credit Item clearing document will be successfully processed with the response code "Successfully completed". 
If the original financial document is not found when processing the clearing document, processing of Notif Prs Credit Item will fail. The warning "Presentment not found for fast funds notification, converting to presentment" will be generated and the clearing document's transaction type will be changed to CH Payment. The next time Document Processing is run to process a clearing document, the updated parameters will be used. 
Issuers offering their cardholders this service are highly recommended to test this functionality online and in clearing. 
In addition, the following checks are required: 
1. Execute the following query and contact Customer Support if the query returns a non-empty result: 
select * from trans_type where amnd_state = 'A' and chain_type = 'X' and prev_trans_type is not null; 
2. Check Our EuroCard/MasterCard and Our Cirrus/Maestro card Service Package settings for trigger conditions in the Service for CH Payment transactions
"DOC_TAG=PTID;DOC_TAG_VALUE=C07,C52,C53,C54,C55,C56,C57,C67;", explicitly identifying express transfer transactions. 
3. Check the settings of menu items for importing VISA and MasterCard clearing information for the presence of the  USE_PAYMENT_TRANSACTION = N parameter. The USE_PAYMENT_TRANSACTION = N parameter is not compatible with  VISA Fast Funds and MasterCard MoneySend, and for payments by Russian state institutions to MIR payment system cards. 
If USE_PAYMENT_TRANSACTION = N is used, contact Customer Support.
